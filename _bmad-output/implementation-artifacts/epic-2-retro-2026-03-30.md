## Deep story analysis — themes

### What went well

1. **Clear pipeline shape** — Planner → Executor → Verifier → raw SQL execution stayed stable across stories; later stories *extended* the route (intent, NULL counts, chart config, follow-ups) without rewriting core safety (NFR7/NFR8).
2. **Prompt registry discipline** — New behaviour (classification, answer, follow-ups, chart) landed as `.txt` prompts, matching FR40 and keeping review focused on wiring, not scattered strings.
3. **Test-driven course corrections** — Story 2.1 documented the `StaticPool` fix for SQLite in-memory tests; that pattern propagated and reduced flaky DB tests in later stories.
4. **Frontend/backend contract fix in 2.3** — `AnalyticsQueryResponse` was corrected early (`columns`/`rows`, `suggested_questions`) before ChatPanel work, avoiding a larger 2.6 firefight.
5. **FR41 isolation (2.7)** — Analytics had no extraction imports already; AST-based tests and `analytics/__init__.py` contract comment lock the boundary without a fake second app in production.

### Challenges and friction (systems, not blame)

1. **LLM call chain length** — Each successful query touches many ModelClient calls (classify, plan, SQL, answer, chart, follow-ups). Tests that use positional `side_effect` lists broke whenever a call was added (called out in 2.2, 2.3, 2.7 dev notes). The *system* needs either clearer mock helpers or a single seam for “pipeline responses.”
2. **Planner gap in 2.4** — `previous_sql` was wired through the route and executor but initially omitted from `planner.plan()` messages. A **contract checklist** (same fields in planner and executor user payloads for shared concepts) would catch this earlier.
3. **Type mismatches from scaffold** — `AnalyticsQueryRequest` briefly had `previous_sql` under `context` in TypeScript; fixed in 2.6. **Source of truth** should remain backend Pydantic schemas with frontend types verified in CI or a small contract test.
4. **Deferred cross-epic debt** — Items explicitly parked for Epic 4+ (multi-table NULL counts), Epic 5 (structured errors, retries, timeouts), and Epic 6 (singleton ModelClient, pooling, auth) are documented in story reviews and `deferred-work.md`; the team should **trace** them when those epics start.

### Technical debt carried forward (tracked)

| Area | Notes |
|------|--------|
| ModelClient lifecycle | Per-request instantiation; pooling/singleton deferred to deployment hardening. |
| Pipeline mock brittleness | `side_effect` length matches call count; fragile when pipeline grows. |
| `_count_null_exclusions` | Assumes `shipments`-centric semantics; superseded by multi-table behaviour in Epic 4. |
| Hook UX | Concurrent `query()`, reset vs in-flight requests, optional hiding of `previousSql` — partially deferred. |
| Executor fence stripping | Preamble before code fences called out in deferred-work for a future LLM-response hardening pass. |

## Key takeaways

1. **Stable agent pipeline + prompt files** scaled better than ad-hoc prompt strings as Epic 2 grew.
2. **Cross-layer contracts** (TypeScript ↔ Pydantic, planner ↔ executor context) need explicit checks to avoid subtle drift.
3. **Test mocks must track LLM call count** — invest in helpers early to reduce churn when the pipeline gains steps.
4. Epic 3 can proceed **without** replanning Epic 2’s scope; linkage work remains correctly placed in Epic 4.
