## 2. What went well

1. **Sequencing matched risk:** Envelope and handlers (5.1) before retry logic (5.2) before transport errors (5.3) before SQL-specific HTTP semantics (5.4) before fallback (5.5) before UI polish (5.6) reduced rework.

2. **Single contract for HTTP errors:** Frontend types and `getErrorResponseFromUnknown` align with backend — no parallel “analytics error string” explosion for the paths migrated in 5.4.

3. **Explicit cross-story coordination** in story files (5.2 vs 5.3 vs 5.5 ordering; 5.4 vs verifier) made ownership of each failure domain clear.

4. **Review feedback was actionable:** Examples: `clear_overrides` in `test_story_5_4.py`; `ErrorToast` `placement` to avoid stacked fixed toasts — small diffs, clear payoff.

5. **FR coverage:** FR29–FR33, NFR11 (structured handling within client timeout budget), UX-DR4 are addressed in code paths documented in story completion notes.

## 3. Challenges and growth areas (systems, not blame)

| Theme | Evidence |
|--------|-----------|
| **Contract migration** | 5.4 moved unsafe SQL from “200 + body error field” to proper non-2xx `ErrorResponse` — correct for clients, but required coordinated test and frontend updates. |
| **Test harness details** | 5.1: `raise_server_exceptions=False` for global 500 handler tests; 5.4: dependency override cleanup — patterns should be copied whenever new FastAPI integration tests land. |
| **Failure domain boundaries** | Broad `SQLAlchemyError` on execute can include infra-like cases (e.g. DB locked) — same envelope as bad SQL for v1; noted for product if distinct `error_type` is ever needed. |
| **Config hygiene** | 5.5: duplicate primary/fallback model IDs in env can cause redundant phases — deferred to optional startup validation. |
| **Optional UX** | Schema fetch after rate limit does not auto-refetch when countdown ends — acceptable stretch; user can refresh. |
